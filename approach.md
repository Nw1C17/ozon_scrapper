## 1. Выбор инструментов

Перед началом разработки проанализировал доступные библиотеки для парсинга веб-страниц.  
С помощью AI составил список по назначению:

| Назначение | Библиотеки |
|------------|------------|
| HTTP‑запросы | `requests`, `httpx` |
| Разбор HTML/XML | `BeautifulSoup`, `lxml` |
| Автоматизация браузера | `Selenium`, `Playwright`, `Pyppeteer` |
| Работа с JavaScript | `Selenium`, `Playwright`, `pyppeteer` |
| Обработка данных | `pandas`, `json`, `csv` |

Из всех вариантов выбрал **Selenium + webdriver-manager** по следующим причинам:

- **Pyppeteer / Puppeteer** — работают только с Chromium, предположил что мне нужна поддержка большего количества браузеров.
- **requests + BeautifulSoup** — не выполняют JavaScript, а Ozon использует динамическую подгрузку товаров. Восстанавливать все AJAX-запросы, подписывать заголовки и обрабатывать динамические токены вручную. Сложна.
- **Playwright** — на момент выбора был менее знаком, к тому же требует установки своих сборок браузеров, которые, как выяснилось позже, имеют специфические признаки в TLS-отпечатке и часто отстают по версиям, что может приводить к детекции WAF. (Эту информацию получил в уточняющем запросе к AI, когда изучал минусы Playwright.)
- **Selenium + webdriver_manager** — по информации ai, медленнее чем Playwright, но я выбрал из соображений надежности.

## 2. Первая реализация парсера (Selenium + Chrome)

### Что пробовал
- Использовал стандартный Selenium WebDriver с Chrome.
- Добавил маскировку `navigator.webdriver` через JavaScript.
- Внедрил ротацию User-Agent.
- Установил случайные задержки между запросами страниц (3–6 сек).
- Реализовал ожидание загрузки элементов (`WebDriverWait`).
- Настроил headless-режим для фоновой работы.

Для выполнения **Части 2** (три последовательных запуска с интервалом 30 секунд) я временно запускал скрипт **в сети без прокси**. При таких интервалах и ротации User-Agent блокировок не наблюдалось, скрипт отработал стабильно.

## 3. Поддержка Firefox и Edge

### Что пробовал
Настроил драйвер Firefox и Edge с аналогичными параметрами (user-agent, отключение WebDriver-флагов через `about:config`, headless). Запустил несколько тестов. Firefox не мог обойти антибот систему

### Возможные причины почему Firefox обходит антибот хуже
Firefox оказался ещё более уязвимым для детекта. Причины:
- **Другая архитектура WebDriver.** В Firefox флаг `dom.webdriver.enabled` можно отключить, но существуют другие сигналы, например, наличие `navigator.webdriver` даже при отключённом флаге в некоторых версиях.
- **Отсутствие тонкой настройки headless-режима.** В Firefox headless-режим менее зрелый и оставляет больше артефактов, которые можно выявить.
- **Headless режим.** - В коде для Firefox используется --headless, а для Chrome --headless=new. Новый режим headless в Chrome (начиная с версии 109) лучше имитирует обычный браузер, в то время как старый headless режим Firefox может быть более детектируемым
- **TLS-отпечаток (JA3) может выдать автоматизацию**

### Попытки решения 
- **Использовал существующий профиль Firefox**
- **Удалить свойство navigator.webdriver через JavaScript (driver.execute_script("Object.defineProperty(navigator, 'webdriver', {get: () => undefined})"))**
- **Эмулировал реальные события мыши и прокрутку**


### Итог
Ни один из этих методов не смог обеспечить стабильную работу для трёх последовательных запусков. Для масштабирования в дальнейшем планирую попробовать proxy-пул.

## 4. Прокси с аутентификацией

### 4.1. Исходная ситуация
При тестировании на рабочем компьютере использовался корпоративный прокси-сервер с NTLM-аутентификацией. Прокси передавался через аргумент `--proxy` в формате `http://user:pass@host:port`. В коде добавлял `--proxy-server=...` в ChromeOptions.

### 4.2. Проблема
Selenium и ChromeDriver **не поддерживают передачу логина и пароля в строке прокси напрямую**. Браузер пытался подключиться, но аутентификация не проходила, в логах появлялась ошибка `407 Proxy Authentication Required`.

### 4.3. Решение
1. **Расширение браузера** — пытался настроить прокси через расширение с автонастройкой, но это требовало установки дополнительных расширений и сильно усложняло код.
2. **Playwright** — рассматривал переход, так как он имеет встроенную поддержку прокси с логином/паролем. Однако отказался из-за необходимости установки его собственных браузеров (что увеличивает размер и время развёртывания), а также из-за риска детекции по TLS-отпечатку (кастомный Chromium). Тесты подтвердили что Playwright не обходит антибот-систему.
3. **Selenium-wire** — для решения проблемы прокси с логином/паролем попробовал использовать библиотеку selenium-wire, которая позволяет передавать аутентификацию прямо в строке прокси, таким образом удалось решить проблему при тестировании, однако решил не использовать ее в итоговой версии поскольку на Windows 10 наблюдалась низкая производительность (запросы выполнялись более 2 минут)
4. **Увеличения ротации**
### Что сделал
В коде оставил возможность передавать прокси через аргумент `--proxy`, но только для прокси без аутентификации.

## 5. Решение через camoufox

### Интеграция camoufox в проект

В процессе поиска решений для firefox, наткнулся на camoufox. Я добавил поддержку `camoufox` как отдельный тип браузера (`--browser camoufox`).
Тесты показали, что с camoufox парсер стабильно отрабатывает три запуска подряд без капчи. Camoufox работает даже без дополнительной маскировки `navigator.webdriver` — он уже патчит его автоматически.

### Что такое camoufox
**Camoufox** — это специализированная модификация Firefox, В отличие от обычного Firefox, camoufox:
- **Патчит внутренние признаки автоматизации.** Удаляет или подменяет свойства `navigator.webdriver`, `navigator.plugins`, `navigator.languages`, имитируя реального пользователя.
- **Использует настоящий движок Firefox**, но с изменёнными настройками профиля, чтобы не оставлять следов Selenium.
- **Работает в headless-режиме** без характерных для обычного headless-режима артефактов (например, сохраняет полную поддержку WebGL, аудио, видео).
- **Эмулирует поведение человека** — скролл, задержки, клики.

### Итог
camoufox смог обеспечить стабильную работу для трёх последовательных запусков.

## 6. Работа с AI

1. **«Составь список основных библиотек для парсинга веб-страниц, разделив их по назначению: HTTP-запросы, разбор HTML/XML, автоматизация браузера, работа с JavaScript, обработка данных.»**  
   Получил структурированный обзор, который помог принять решение в пользу Selenium.

2. **«Напиши парсер на Python для страницы товара на Ozon (например, https://www.ozon.ru/product/...). Необходимо извлечь артикул (SKU) товара. Учти, что данные могут подгружаться динамически. Используй Selenium. Добавь обработку ошибок, задержки и ротацию User-Agent.»**  
   AI сгенерировал базовый код с извлечением SKU через регулярное выражение. Однако регулярка исключала все цифры, оставляя пустую строку. Вручную исправил на  
   `r'/product/(?:[a-z0-9-]+-)?(\d+)'`  
   Также увеличил задержки: вместо фиксированного `time.sleep(2)` добавил `random.uniform(3, 6)` и `random.uniform(1, 3)` между действиями.

3. **«Сгенерируй список из 20 User-Agent…»**  
   AI выдал 20 строк, но для тестового запуска ограничился 5 самыми, чтобы не усложнять код. В будущем планирую использовать полный список.  
   Также AI предложил ротировать заголовки (`Accept-Language`, `Referer`). Реализовал выбор случайного `User-Agent` при каждом запуске, в перспективе добавлю и ротацию других заголовков.

4. **«Напиши руководство по использованию Selenium с прокси с аутентификацией»**  
   AI объяснил, что стандартные опции Chrome не поддерживают `user:pass@host:port`, и предложил два пути: `selenium-wire` и переход на Playwright.  
   Оба варианта протестировал, но они не сработали в моей среде (см. раздел о прокси).

## 8. Выводы и дальнейшие улучшения

- Основная проблема — прокси с аутентификацией — требует отдельного решения. Для масштабирования, планируется доработка selenium-wire.
- AI помог быстро получить рабочий прототип, но потребовалась доработка
- Для повышения устойчивости в архитектуре планируется добавить:
  - пул прокси с ротацией,
  - интеграцию с сервисом распознавания капчи,
  - распределение задач через очередь (Celery).
